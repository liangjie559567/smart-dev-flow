# 接收代码审查

## 核心原则

**先验证，后实施。技术正确性优先于社交舒适度。**

## 响应模式

```
1. 读取：完整阅读反馈，不急于反应
2. 理解：用自己的话复述需求（或提问）
3. 验证：对照代码库现实检查
4. 评估：对当前代码库技术上是否合理？
5. 响应：技术确认或有理由的反驳
6. 实施：逐项修复，每项测试
```

## 禁止回应

**绝对不说：**
- "你说得完全正确！"（表演性同意）
- "好建议！"（表演性）
- "让我现在实现它"（验证前）

**应该说：**
- 复述技术需求
- 提出澄清问题
- 用技术理由反驳（如果有误）
- 直接开始修复（行动 > 语言）

## 不清晰时

```
如果任何条目不清晰：
  停止 - 不实施任何内容
  先澄清所有不清晰的条目

原因：条目可能相关。部分理解 = 错误实施。
```

## 实施顺序

1. 先澄清所有不清晰的条目
2. 按优先级实施：
   - 阻塞性问题（崩溃、安全）
   - 简单修复（拼写、导入）
   - 复杂修复（重构、逻辑）
3. 每项单独测试
4. 验证无回归

## 何时反驳

- 建议破坏现有功能
- 审查员缺乏完整上下文
- 违反 YAGNI（未使用的功能）
- 对当前技术栈技术上不正确
- 与架构决策冲突

**如何反驳：** 使用技术理由，不是防御性的。引用可运行的测试/代码。

## 确认正确反馈时

```
✅ "已修复。[简短描述变更内容]"
✅ "发现了 - [具体问题]。已在 [位置] 修复。"
✅ [直接修复并在代码中展示]

❌ "你说得对！"
❌ "好发现！"
❌ 任何感谢表达
```

## 与 Axiom 集成

- 在 `axiom-implement` 代码审查阶段收到 REQUEST CHANGES 后调用
- 修复完成后重新运行 `verification-before-completion`
- 验证通过后更新 `active_context.md`：`task_status: REFLECTING`
