# 头脑风暴 - 将想法转化为设计

## 概述

通过自然协作对话，将想法转化为完整的设计和规格。

先了解当前项目上下文（读取 `.agent/memory/active_context.md`），然后逐一提问细化想法。理解要构建的内容后，展示设计并获得用户批准。

<HARD-GATE>
在展示设计并获得用户批准之前，不得调用任何实现技能、编写任何代码、搭建任何项目或采取任何实现行动。无论项目看起来多简单，此规则适用于所有项目。
</HARD-GATE>

## 检查清单

必须按顺序完成以下每项任务：

1. **探索项目上下文** — 读取 `.agent/memory/active_context.md`、相关文档、最近提交
2. **逐一提问** — 每次只问一个问题，理解目的/约束/成功标准
3. **提出 2-3 个方案** — 含权衡分析和推荐理由
4. **展示设计** — 按复杂度分节展示，每节后询问是否正确
5. **编写设计文档** — 保存到 `docs/plans/YYYY-MM-DD-<主题>-design.md` 并提交
6. **更新 active_context.md** — 将 `task_status` 更新为 `DRAFTING`
7. **过渡到实现** — 调用 `writing-plans` 技能创建实现计划

## 流程

```
探索项目上下文
  ↓
逐一提问（每次只问一个）
  ↓
提出 2-3 个方案（含权衡和推荐）
  ↓
分节展示设计 → 用户批准？
  ├─ 否（修改）→ 重新展示
  └─ 是 → 编写设计文档 → 调用 writing-plans
```

**终态是调用 writing-plans。** 不得调用其他实现技能。

## 核心原则

- **每次只问一个问题** — 不要用多个问题压倒用户
- **优先多选题** — 比开放式问题更易回答
- **YAGNI 原则** — 从所有设计中删除不必要的功能
- **探索替代方案** — 在确定方案前始终提出 2-3 个选项
- **渐进式验证** — 展示设计，获得批准后再继续

## 与 Axiom 状态机集成

- 设计批准后：将 `task_status` 更新为 `DRAFTING`，写入 `session_name`
- 调用 `writing-plans` 生成实现计划
- 计划完成后：调用 `axiom-draft` 进入 Phase 1

## 禁止行为

- 在设计批准前编写任何代码
- 跳过方案比较直接进入实现
- 一次提多个问题
