# 使用技能系统

<EXTREMELY-IMPORTANT>
如果你认为某个技能有哪怕 1% 的可能适用于当前任务，你绝对必须调用该技能。

如果技能适用于你的任务，你没有选择。你必须使用它。

这不可协商。这不是可选的。你无法通过合理化来回避这一点。
</EXTREMELY-IMPORTANT>

## 规则

**在任何响应或行动之前，先调用相关技能。** 即使只有 1% 的可能性适用，也要调用技能检查。

## 技能优先级

1. **流程技能优先**（brainstorming、systematic-debugging）— 决定如何处理任务
2. **实现技能其次**（test-driven-development、executing-plans）— 指导执行

## 技能类型

**刚性技能**（TDD、调试）：严格遵循，不要因为"这次不同"而跳过。

**灵活技能**（模式类）：根据上下文调整原则。

## 红旗警告

这些想法意味着停止——你在合理化：

| 想法 | 现实 |
|------|------|
| "这只是个简单问题" | 问题也是任务。检查技能。 |
| "我需要先了解更多上下文" | 技能检查在澄清问题之前。 |
| "这不需要正式技能" | 如果技能存在，就使用它。 |
| "我记得这个技能" | 技能会演进。读取当前版本。 |
| "技能太重了" | 简单的事情会变复杂。使用它。 |

## 与 smart-dev-flow 集成

- 技能通过 `skill-injector.mjs` 自动注入（基于 triggers 关键词匹配）
- 会话启动时 `session-start.cjs` 根据 Axiom 状态推荐相关技能
- 使用 Skill 工具调用技能，不要用 Read 工具读取技能文件
